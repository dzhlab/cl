Создай подробную документацию для $1

## Типы документации:

### 1. API документация
Для каждой публичной функции/метода/класса создай:

```javascript
/**
 * Краткое описание функции
 *
 * @param {type} paramName - Описание параметра
 * @param {type} [optionalParam] - Необязательный параметр
 * @returns {type} Описание возвращаемого значения
 * @throws {ErrorType} Когда и почему выбрасывается
 * @example
 * // Пример использования
 * const result = functionName(param1, param2);
 */
```

### 2. README для модуля/компонента
Включи:
- **Назначение** - что делает модуль
- **Установка** - как установить зависимости
- **Использование** - примеры кода
- **API** - список функций с описанием
- **Конфигурация** - если есть настройки
- **Примеры** - реальные use cases
- **Ограничения** - что модуль НЕ делает
- **FAQ** - частые вопросы

### 3. Архитектурная документация
Опиши:
- Структуру файлов и директорий
- Зависимости между модулями
- Паттерны проектирования
- Принятые архитектурные решения
- Диаграммы (если применимо)

### 4. Inline комментарии
Добавь комментарии к:
- Сложной бизнес-логике
- Неочевидным решениям
- Workarounds и temporary solutions
- Важным константам

## Формат:

### Для JavaScript/TypeScript:
- JSDoc комментарии
- Markdown README
- TypeScript types как документация

### Для Python:
- Docstrings (Google/NumPy/Sphinx стиль)
- Type hints
- Markdown README

### Для Go:
- Godoc комментарии
- Package documentation
- Examples в тестах

### Для Java:
- Javadoc
- Markdown README

## Требования:

✅ Понятный язык (без жаргона где возможно)
✅ Примеры кода для каждой функции
✅ Описание параметров и возвращаемых значений
✅ Указание возможных ошибок
✅ Актуальность (соответствие коду)

❌ Не описывай очевидные вещи
❌ Не дублируй информацию
❌ Не создавай документацию "для галочки"

## Действия:

1. Прочитай код в $1
2. Определи публичный API
3. Добавь JSDoc/Docstring комментарии
4. Создай или обнови README.md если нужно
5. Добавь примеры использования
6. Убедись что примеры рабочие (можно запустить)
